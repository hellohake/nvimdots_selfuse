# hello-spec-start

## Purpose

Start a `hello-spec-v2` proposal without fast-forwarding past human gates.

This is the preferred entrypoint for new `hello-spec-v2` changes. It replaces the
pattern of manually calling `openspec-new-change` and then accidentally using
`openspec-ff-change`.

## Input Shape

Accept natural language, not CLI-style flags:

```text
hello-spec-start <change-name> <initial user input...>
```

Examples:

```text
hello-spec-start p3-pack-card-runtime-redesign 基于飞书技术方案 https://...，重做 P3 C端打包运行时；base=feat/card_type_support；不要复用旧实现计划。
```

```text
hello-spec-start p3-pack-card-runtime-redesign
```

If only the change name is provided, stop and ask for the initial input source:

```text
请提供这次提案的原始输入：可以是飞书文档链接、本地文档路径，或直接粘贴需求。
```

Do not require `--prompt` or other flags.

## Source Input Policy

The initial user input is an early-stage source. It is important before the
technical plan has converged, but it should not keep overriding canonical
artifacts later.

Why this matters:

- If long prompts or external design docs are only kept in chat, `/clear`,
  fast-forward attempts, or a new agent can lose key source constraints.
- If raw prompts are still treated as primary input during tasks/plan, they add
  noise and can override decisions already settled in proposal/design/grill-spec.

Rules:

- Do NOT create `source.md`.
- Do NOT write `intake.md`; `intake.md` is generated by retrospective workflows
  such as `spec-opti-workflow`.
- High-density initial input must have a re-readable external source when it
  contains any of:
  - Lark/Feishu/doc URL;
  - local file path;
  - more than 500 characters;
  - explicit hard constraints that would be expensive to lose.
- For a short inline prompt without external source, record the raw input
  summary in `brainstorm.md`.
- If high-density input has no readable source path/link, stop and ask the user
  to provide one, or explicitly confirm degraded mode. On degraded mode, record
  `NO_SOURCE_CONFIRMED=true` in the output and in `brainstorm.md` Input Sources.
- Consume the initial input when creating `brainstorm.md` and `proposal.md`.
- For `specs` / `design`, consume `proposal.md` and `brainstorm.md` first; only
  re-read the original source reference when checking scope, resolving ambiguity,
  or validating that a hard user constraint was not lost.
- For `tasks` / `plan`, do NOT use raw initial input as the normal source of
  truth. Use canonical artifacts (`proposal.md`, `specs/**/*.md`, `design.md`,
  `grill-spec.md`). If those artifacts conflict with initial input, stop and
  route through `spec-plan-revise`.

When creating `brainstorm.md`, include a small visible source trace:

```markdown
## Input Sources

- Primary input: <pasted in start turn | lark/doc url | local path>
- Source consumed: yes
- Source summary: <short summary>
- Hard constraints extracted: <short list>
```

This trace is enough for later phases to know where the source came from without
adding another mandatory file.

## Workflow

### Phase 0 - Identity Lock

Start with:

```text
已锁定任务：hello-spec-start；本轮只启动 hello-spec-v2 提案并推进到第一个业务 artifact，不进入 apply，不改业务代码。
```

Forbidden objectives:

- Do not run `openspec-propose`.
- Do not run `openspec-ff-change`.
- Do not run `openspec-apply-change`.
- Do not generate multiple business artifacts in one turn.
- Do not create `source.md`.
- Do not edit business source code.
- Do not run tests/builds.
- Do not commit, push, or archive.

### Phase 1 - Parse Request

Extract:

- `CHANGE_NAME`: the first kebab-case token after `hello-spec-start`, or infer a
  kebab-case name from the user's request if no explicit name exists.
- `INITIAL_INPUT`: all remaining user text after the change name.

If `CHANGE_NAME` is unclear, ask for it. If `INITIAL_INPUT` is empty, ask for the
initial input source and stop.

### Phase 1.2 - Source Input Gate

Run this before creating the OpenSpec change, so an unreadable source does not
leave a half-initialized change directory.

Classify `INITIAL_INPUT`:

- `short_inline`: short direct request, no external source needed.
- `external_source`: contains a Lark/Feishu/doc URL or local file path.
- `high_density_without_source`: more than 500 characters or contains hard
  constraints, but no readable URL/path.

Rules:

- `short_inline`: continue. Later write the raw input summary into
  `brainstorm.md` Input Sources.
- `external_source`: verify the URL/path is readable enough to consume. If it
  cannot be read, stop and ask:

  ```text
  1. 补充可读路径/链接
  2. 降级继续（NO_SOURCE_CONFIRMED=true）
  ```

- `high_density_without_source`: stop and ask for the same two choices. Do not
  treat pasted content as "good enough" for recovery. If the user confirms
  degraded mode, continue and record `NO_SOURCE_CONFIRMED=true`.

Do not write a new source file. Do not write `intake.md`.

### Phase 1.5 - Schema Discovery

Before creating the change, confirm this repo is configured for `hello-spec-v2`.

1. Locate the repo root from cwd. Prefer the nearest directory containing
   `openspec/`; otherwise use the current git root.
2. Read `.openspec.yaml` if present.
3. Determine the selected/default schema. If `.openspec.yaml` or `openspec`
   status/instructions cannot confirm `hello-spec-v2`, stop and report:
   - cwd;
   - repo root candidate;
   - discovered config path;
   - discovered schema/default schema if any;
   - the command the user should run or the repo path they should provide.

Do not create a non-`hello-spec-v2` change and then discover the mismatch after
the fact.

### Phase 2 - Create Change

Before running `openspec new change`, check whether
`openspec/changes/<CHANGE_NAME>` already exists.

Existing change handling:

- If it exists and `openspec status --change "<CHANGE_NAME>" --json` succeeds
  with `schemaName=hello-spec-v2`, do not recreate or overwrite it. Stop and
  report that the change already exists, then suggest:

  ```text
  hello-spec-next <CHANGE_NAME>
  ```

- If it exists and status shows partial artifacts, treat it as an existing
  change, not a start target. Stop and hand off to `hello-spec-next`.
- If it exists but schema is not `hello-spec-v2`, stop and report the discovered
  schema.
- If the directory exists but OpenSpec status is unreadable, stop and report the
  path plus the status error. Ask the user whether to inspect/repair manually.

Only create a new change when no existing change directory/status is present.

Run:

```bash
openspec new change "<CHANGE_NAME>"
```

Use the repository's default schema. If the resulting schema is not
`hello-spec-v2`, stop and explain that this skill is only for `hello-spec-v2`.

Then run:

```bash
openspec status --change "<CHANGE_NAME>" --json
```

Confirm `schemaName` is `hello-spec-v2`.

### Phase 3 - Placeholder Fast Path

Use this explicit algorithm. Do not invent placeholder behavior.

Loop:

1. Run:

   ```bash
   openspec status --change "<CHANGE_NAME>" --json
   ```

2. Find the first `ready` artifact.
3. If it is not one of these placeholder IDs, stop the loop:
   - `troubleshoot`
   - `revise`
   - `manual-test-commands`
4. For a placeholder, run:

   ```bash
   openspec instructions <artifact-id> --change "<CHANGE_NAME>" --json
   ```

5. Read `template` and `resolvedOutputPath` from the JSON.
6. Write the template verbatim to `resolvedOutputPath`.
7. Re-run status and continue the loop.

Never infer placeholder content. Never create business artifacts in this phase.
Stop when the next ready artifact is business-bearing or a gate.

### Phase 4 - First Business Artifact

If the next ready artifact is `brainstorm`, the start invocation with non-empty
`INITIAL_INPUT` counts as permission to draft exactly this one artifact unless
the user explicitly said "only create the change". After creating brainstorm,
stop.

Rules for `brainstorm.md`:

- If input is `short_inline`, consume the current-turn input directly.
- If input is `external_source`, read the URL/path with the relevant available
  tools/skills before drafting.
- If degraded mode was confirmed, consume the current-turn pasted input and mark
  `NO_SOURCE_CONFIRMED=true`.
- Include `## Input Sources` as described above.
- Extract goals, non-goals, hard constraints, known references, and open questions.
- Do not create `proposal.md` in the same turn unless the user explicitly asks
  after reviewing brainstorm.

If the next artifact is not `brainstorm`, stop and show the next instruction.

### Phase 5 - Output Format

Finish with:

````markdown
## Status
- Change:
- Schema:
- Progress:
- Initial input:
- Source input:
- Auto-created placeholders:
- Stopped at:

## Why Stopped
<one short explanation>

## Next Command
```text
hello-spec-next <change>
```

## Suggested /clear Resume
```text
cd <repo root> && hello-spec-next <change>
```
````

## Success Criteria

- A new `hello-spec-v2` change exists.
- Initial input was consumed into `brainstorm.md` when brainstorm was created, or
  the skill stopped and asked for input.
- Only lightweight placeholders were auto-created.
- No `source.md` or `intake.md` was created.
- The workflow stopped before proposal/specs/design/tasks/plan unless explicitly
  asked to draft exactly one ready business artifact.
