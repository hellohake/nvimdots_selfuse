# 对话归档与深度分析

本技能要把当前上下文里的问题排查 / 技术讨论**重新组织、深度整合**，捕捉用户触发时附带的新问题并结合上下文给出有分量的解答，最终沉淀为结构化、可复用的 Markdown 文档。

**整合而非记录。** 对话是发散、跳跃、带回退的，把聊天记录直接倒进文件毫无价值；读者要的是去噪后的清晰判断——现象、根因、最终方案、为什么这样选——目的是让后来者复用判断，而不是重放聊天现场。

**动手写之前，先把每条信息归到四类之一。** 归档前先自问：

- 哪些是**已证实的事实**（命令输出 / 日志 / 代码 / 工具结果）？
- 哪些只是**合理推断**（必须写清依据）？
- 哪些是**证据支撑的结论**（才配进 Root Cause / 最终结论）？
- 哪些**仍待验证**（只能进"待验证假设 / 遗留问题"）？

分清这四类是整篇归档可信度的地基，混淆它们正是低质量归档的根源。每类的落地处理见阶段 2.5。

## 解析用户输入

用户触发时给的参数可能包含两类内容，需要分辨：

1. **目标目录**：形如 `docs/`、`./.ai_doc`、某个绝对路径。识别为归档位置。
2. **追问 / 分析指令**：形如"分析下高并发会怎样""顺便对比方案 B""为什么不用 X"。识别为需要在阶段 3 深度处理的分析任务。

两类可能同时出现，也可能都没有。没有目录时用默认归档位置，没有追问时跳过深度分析章节。

## 工作流

### 阶段 1：目录预检与确认（写文件前的硬约束）

1. **定落点**：有显式目录就用它，且不强行追加 `summary/YYYYMM/`；没有则默认 `.ai_doc/summary/YYYYMM/`（如 `.ai_doc/summary/202606/`）。用户只写 `.ai_doc/` 时，提示将落到 `summary/YYYYMM/` 子目录。先读仓库 `AGENTS.md` / `CLAUDE.md`，其归档目录 / 语言 / 格式 / 敏感信息约束优先。
2. **拦截（硬约束）**：`.ai_doc/` 常是软链接，目录"看起来不存在"时先确认有没有指向它的软链接再判断。若根目录或用户指定目录确认不存在，**先停下、不要写文件**，向用户确认后再继续——不确定落点就写，内容会散落到错误位置：
   > "未找到目标目录 `{目录名}`。需要我创建它吗？或者你也可以指定一个已存在的目录。"
3. **落点确定后**：可自动创建缺失的 `summary/` 和 `YYYYMM/` 子目录（前提是 `.ai_doc/` 已存在），创建前说明最终目录；文件重名时追加 `-HHMM` 或更具体主题词。除非用户显式给了根目录，否则绝不把文档平铺进 `.ai_doc/` 根目录——根目录会迅速被归档文件淹没、再也找不到；覆盖旧归档同理见 NEVER。

### 阶段 2：归纳与结构化

判断本次对话属于哪种类型，再套用「文档结构模板」里对应的骨架整理核心草稿：

- **排查记录 (Troubleshooting)**：用排查类骨架。
- **技术讨论 (Discussion)**：用讨论类骨架。

如果一次对话两者兼具，以主线为准，另一部分作为子章节并入。

若上下文太短、为空、或全是互不相关的零碎话题，不足以构成一份有价值的归档，先告知用户并询问是否仍要生成最小骨架，而不是硬凑一篇文档。

### 阶段 2.5：证据分层落地（四类定义见开头思维框架）

四类的写法各不相同，落笔时让读者一眼看出每条信息的证据强度：

- **事实** → 直接进正文，保留关键路径、命令、错误码、时间点或文件位置。
- **推断** → 必须带依据，写成"因为 A 日志出现在 B 操作之后，所以推断…"。
- **结论** → 只有它能进入 `Root Cause` / `最终结论`。
- **待验证** → 不得伪装成结论，只能进"待验证假设 / 后续验证 / 风险与遗留问题"。

正文用标注体现强度，例如：

```markdown
## 证据与事实
- [事实] 错误日志 `dial tcp: i/o timeout` 出现在 14:02，紧接一次发布之后（来自用户贴的日志）。
- [推断] 超时集中在发布后 → 怀疑新版本连接池配置变更引入，依据是时间相关性，尚未看代码。
- [待验证] 是否为 `MaxIdleConns` 调小所致，需 diff 发布前后配置确认。
```

如果上下文不足以支撑 Root Cause，不要强行写"根本原因"。改用 `当前结论 / 待验证假设`，明确说明哪些部分已经确认，哪些仍需要日志、代码、复现或用户补充验证。

每份归档都要保留最终状态：`已解决`、`部分解决`、`未解决`、`仍待验证` 四选一或组合说明。状态比过程更重要，因为读者通常先判断这份文档能不能直接指导下一步行动。

### 阶段 2.6：图示规则

归档默认面向终端和纯 Markdown 阅读环境。当图能显著降低理解成本时再画图，不要为了装饰而画。所有图示必须使用 Markdown 代码块里的 ASCII 图——这样读者不依赖 Mermaid / PlantUML / Graphviz 等渲染器也能直接看懂：

```text
User Request
     |
     v
Collect Facts ---> Separate Evidence ---> Write Conclusion
     |                    |                    |
     v                    v                    v
Known Gaps         Inferences Labeled     Final Status
```

适合使用 ASCII 图的场景：

- **流程图**：请求链路、排查路径、决策步骤。
- **时序图**：多个模块或角色之间的调用顺序。
- **结构图**：组件依赖、数据流、分层关系。

图示要短、横向可读、节点名称来自真实上下文。若图太复杂，拆成多个小图或改成表格。

### 阶段 3：动态追问与深度分析（关键环节）

1. 检查参数里是否带了追问或额外分析要求。
2. 若有，**基于当前上下文做完整技术推理并给出解答**，不要只复述问题。这是文档增值的地方——用户专门提的问题，往往是他真正关心、但对话里还没讲透的点。
3. 把用户的追问消化成分析结论、方案权衡或待验证问题，准备并入最终文档；不要把追问原文当作章节内容。
4. 若追问依赖缺失上下文，先写明"可归档范围"和"缺口"，不要补写不存在的排查过程。

### 阶段 4：生成与输出

1. 综合阶段 2、3 生成最终 Markdown；触发了深度分析时必须新增 `## 扩展分析与延伸探讨`（或更精准的标题）承载解答。
2. 文件命名：`troubleshoot_{YYYYMMDD}_{简述}.md` 或 `discussion_{YYYYMMDD}_{简述}.md`，简述用简短中文或英文短横线词组。
3. 正文只保留能解释问题、决策或后续动作的信息；失败尝试、工具噪声、闲聊、重复确认默认删掉，除非它们解释了关键决策或排除路径。
4. 写入后自检并向用户回报：确认标题 / 日期 / 核心章节 / 最终状态齐全、图示符合阶段 2.6；回复里给出归档路径、文档类型、核心主题、最终状态（按用户指定目录写入则说明覆盖了默认策略），不要重复整篇文档。

## NEVER（硬底线 + 踩坑后果；判定细节见对应阶段，不重述）

- NEVER 直接复制聊天流水账或原样照抄用户追问——照抄等于没整合，读者拿不到去噪后的判断，文档失去全部复用价值。
- NEVER 把没有证据支撑的猜测写成 Root Cause——后来者会把推断当定论照做，错误判断被放大传播。
- NEVER 省略最终状态——读者第一眼就要判断"能不能据此行动"，无状态归档形同废纸。（四状态取值见阶段 2.5）
- NEVER 在落点未确认（目录不存在、软链接未排查）时写文件——内容会散落到错误位置且事后难发现。（判定见阶段 1）
- NEVER 覆盖已有归档文件，除非用户明确要求——旧归档被盖无法找回。
- NEVER 写入与本次归档无关的敏感信息、凭证、个人隐私或内部噪声——归档常被分享/索引，泄露面远大于一次聊天。
- NEVER 使用需额外渲染器的图示格式（Mermaid/PlantUML/Graphviz）——终端与纯文本环境下读者只会看到一坨源码。（格式见阶段 2.6）

## 文档结构模板

骨架是起点不是枷锁：内容侧自由度高，按实际材料增删章节、合并或重命名标题（例如根因不明就把"根本原因"改成"当前结论 / 待验证假设"）。但**最终状态**和证据分层标注必须保留——这两项是归档可信度的底线。

排查类用这个骨架：

```markdown
# [一句话标题]

> 日期：YYYY-MM-DD

## 现象描述
## 证据与事实
## 排查过程
## 根本原因 (Root Cause)
## 最终解决方案
## 最终状态
## 后续验证 / 遗留问题
## 扩展分析与延伸探讨   ← 仅在有追问时出现
```

讨论类用这个骨架：

```markdown
# [一句话标题]

> 日期：YYYY-MM-DD

## 讨论背景
## 已确认事实
## 核心观点与方案权衡
## 最终结论 / Action Item
## 最终状态
## 后续验证 / 遗留问题
## 扩展分析与延伸探讨   ← 仅在有追问时出现
```
