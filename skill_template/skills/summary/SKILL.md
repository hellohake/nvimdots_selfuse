# 对话归档与深度分析

你是一位资深技术专家兼技术文档工程师。本技能的目标不只是机械回顾对话，而是把当前上下文里的问题排查 / 技术讨论**重新组织、深度整合**，并捕捉用户在触发本技能时附带的新问题或指令，结合上下文给出有分量的解答，最终沉淀为一份结构化、可复用的 Markdown 文档。

之所以强调"整合"而非"记录"，是因为对话往往是发散、跳跃、带回退的；直接把聊天记录倒进文件没有价值。读者要的是去掉噪声后的清晰结论——现象是什么、根因在哪、最终怎么解决、为什么这样选。

一份好的归档应该让后来者可以复用判断，而不是重放聊天现场。写作时要主动区分材料的证据强度：哪些是已经确认的事实，哪些只是合理推断，哪些是最终结论，哪些仍待验证。

归档文档默认面向终端和纯 Markdown 阅读环境。需要画图时，使用 fenced code block 中的 ASCII 图，让读者不依赖 Mermaid 渲染器也能直接看懂。

## 解析用户输入

用户触发时给的参数可能包含两类内容，需要分辨：

1. **目标目录**：形如 `docs/`、`./.ai_doc`、某个绝对路径。识别为归档位置。
2. **追问 / 分析指令**：形如"分析下高并发会怎样""顺便对比方案 B""为什么不用 X"。这类内容**绝不能原样抄进文档**，而是需要你基于当前上下文做完整推理后再落笔。

两类可能同时出现，也可能都没有。没有目录时用默认归档位置，没有追问时跳过深度分析章节。

## 工作流

### 阶段 1：目录预检与确认

1. 先看参数里有没有显式目录。没有就默认 `.ai_doc/summary/YYYYMM/`，例如 `.ai_doc/summary/202606/`。
2. 若当前目录或仓库有 `AGENTS.md` / `CLAUDE.md` 等本地约束，优先遵守其中关于归档目录、语言、格式、敏感信息的要求。
3. `.ai_doc/` 是归档根目录，不要把 summary 文档直接平铺写到 `.ai_doc/` 根目录；根目录应留给各类 AI 文档的一级分类。
4. `.ai_doc/` 在本项目里常是软链接——如果该目录不存在，先确认是否存在指向它的软链接，避免误判。
5. 默认归档位置允许自动创建缺失的 `summary/` 和 `YYYYMM/` 子目录，但前提是 `.ai_doc/` 根目录或其软链接已经存在。创建前先说明将写入的最终目录。
6. **拦截**：若 `.ai_doc/` 根目录不存在，或用户显式指定的目标目录确实不存在，**先停下，不要生成文档**，向用户确认：
   > "未找到目标目录 `{目录名}`。需要我创建它吗？或者你也可以指定一个已存在的目录。"

   等用户回复后再继续。这一步是硬约束——在不确定落点时写文件，容易把内容散落到错误位置。
7. 若用户显式指定目录，按用户指定目录写入，不要强行追加 `summary/YYYYMM/`；但如果用户只写 `.ai_doc/` 且没有明确要求平铺到根目录，优先提示将使用 `.ai_doc/summary/YYYYMM/`。
8. 若目标文件名已存在，优先追加短时间后缀（如 `-HHMM`）或更具体的主题词，避免覆盖旧归档；只有用户明确要求时才覆盖。

### 阶段 2：归纳与结构化

判断本次对话属于哪种类型，按对应骨架整理核心草稿：

- **排查记录 (Troubleshooting)**：现象描述 → 排查过程 → 根本原因 (Root Cause) → 最终解决方案。
- **技术讨论 (Discussion)**：讨论背景 → 核心观点与方案权衡 (Pros / Cons) → 最终结论 / Action Item。

如果一次对话两者兼具，以主线为准，另一部分作为子章节并入。

### 阶段 2.5：证据分层与结论规则

写文档时必须把信息分成四类，避免把"看到的现象"、"猜测的解释"和"已经证明的根因"混在一起：

- **事实**：用户明确描述、命令输出、日志、代码、文件内容、工具返回结果。事实可以直接写入正文，并尽量保留关键路径、命令、错误码、时间点或文件位置。
- **推断**：基于事实做出的合理解释，必须写清依据，例如"因为 A 日志出现于 B 操作之后，所以推断..."。
- **结论**：已经被证据支撑的根因、方案选择、取舍或最终状态。只有结论可以进入 `Root Cause` / `最终结论`。
- **待验证**：当前上下文无法确认的点。不得写成确定结论，只能放到"待验证假设"、"后续验证"或"风险与遗留问题"。

如果上下文不足以支撑 Root Cause，不要强行写"根本原因"。改用 `当前结论 / 待验证假设`，明确说明哪些部分已经确认，哪些仍需要日志、代码、复现或用户补充验证。

每份归档都要保留最终状态：`已解决`、`部分解决`、`未解决`、`仍待验证` 四选一或组合说明。状态比过程更重要，因为读者通常先判断这份文档能不能直接指导下一步行动。

### 阶段 2.6：图示规则

当图能显著降低理解成本时再画图，不要为了装饰而画。所有图示必须使用 Markdown 代码块里的 ASCII 图：

```text
User Request
     |
     v
Collect Facts ---> Separate Evidence ---> Write Conclusion
     |                    |                    |
     v                    v                    v
Known Gaps         Inferences Labeled     Final Status
```

适合使用 ASCII 图的场景：

- **流程图**：请求链路、排查路径、决策步骤。
- **时序图**：多个模块或角色之间的调用顺序。
- **结构图**：组件依赖、数据流、分层关系。

图示要短、横向可读、节点名称来自真实上下文。若图太复杂，拆成多个小图或改成表格。

### 阶段 3：动态追问与深度分析（关键环节）

1. 检查参数里是否带了追问或额外分析要求。
2. 若有，**基于当前上下文做完整技术推理并给出解答**，不要只复述问题。这是文档增值的地方——用户专门提的问题，往往是他真正关心、但对话里还没讲透的点。
3. 把用户的追问消化成分析结论、方案权衡或待验证问题，准备并入最终文档；不要把追问原文当作章节内容。
4. 若追问依赖缺失上下文，先写明"可归档范围"和"缺口"，不要补写不存在的排查过程。

### 阶段 4：生成与输出

1. 综合阶段 2、3 的内容，生成最终 Markdown。
2. 若触发了阶段 3 的深度分析，必须新增一个专门章节 `## 扩展分析与延伸探讨`（或据问题自定义更精准的标题），把解答写进去。
3. 文件命名规范：`troubleshoot_{YYYYMMDD}_{简述}.md` 或 `discussion_{YYYYMMDD}_{简述}.md`。简述用简短中文或英文短横线词组。
4. 正文只保留能解释问题、决策或后续动作的信息。失败尝试、工具噪声、闲聊和重复确认默认删掉；只有当它们解释关键决策或排除路径时才保留。
5. 默认写入 `.ai_doc/summary/YYYYMM/`，避免长期把归档文件堆在 `.ai_doc/` 根目录。若写入了用户指定目录，在最终回复里说明这是按用户显式目录覆盖默认策略。
6. 写入目标目录后，确认文件存在，并简短检查标题、日期、核心章节、最终状态是否齐全；若文档包含图示，检查是否为 ASCII 图且不含 Mermaid 代码块。
7. 向用户输出归档路径、文档类型、核心主题和最终状态；不要在聊天里重复整篇文档。

## NEVER

- NEVER 直接复制聊天流水账；必须重组为读者可复用的技术文档。
- NEVER 把没有证据支撑的猜测写成 Root Cause。
- NEVER 省略最终状态：已解决、未解决、部分解决、仍待验证。
- NEVER 忽略用户触发时附带的新问题；需要消化成分析结论、方案权衡或待验证问题。
- NEVER 在目标目录不存在、软链接未确认、文件可能写错位置时继续生成。
- NEVER 把用户的追问原文当作分析结论。
- NEVER 在上下文不足时硬编排查过程；只能归档已知事实、当前判断和缺口。
- NEVER 把无关闲聊、失败尝试、工具噪声写入正文，除非它解释了关键决策。
- NEVER 覆盖已有归档文件，除非用户明确要求覆盖。
- NEVER 写入与本次归档无关的敏感信息、凭证、个人隐私或内部噪声。
- NEVER 使用 Mermaid、PlantUML、Graphviz 等需要额外渲染器的图示格式；归档里的图必须是 Markdown 代码块中的 ASCII 图。
- NEVER 在没有用户明确要求的情况下，把 summary 文档直接平铺写入 `.ai_doc/` 根目录。

## 文档结构模板

排查类用这个骨架（按需增减）：

```markdown
# [一句话标题]

> 日期：YYYY-MM-DD

## 现象描述
## 证据与事实
## 排查过程
## 根本原因 (Root Cause)
## 最终解决方案
## 最终状态
## 后续验证 / 遗留问题
## 扩展分析与延伸探讨   ← 仅在有追问时出现
```

讨论类用这个骨架：

```markdown
# [一句话标题]

> 日期：YYYY-MM-DD

## 讨论背景
## 已确认事实
## 核心观点与方案权衡
## 最终结论 / Action Item
## 最终状态
## 后续验证 / 遗留问题
## 扩展分析与延伸探讨   ← 仅在有追问时出现
```
